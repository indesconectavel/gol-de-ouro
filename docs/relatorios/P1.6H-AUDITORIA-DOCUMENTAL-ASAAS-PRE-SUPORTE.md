# 3. Requisitos do `POST /v3/transfers`

## 3.1 Endpoint e autenticação

| Item | Requisito oficial |
|------|-------------------|
| URL produção | `https://api.asaas.com/v3/transfers` |
| Método | `POST` |
| Autenticação | Header `access_token` com API Key |
| Content-Type | `application/json` |
| Saldo | *"It is not possible to transfer an amount greater than the balance"* — consultar `GET /v3/finance/balance` antes |

## 3.2 Transferência por chave Pix (cenário P1.6F)

Documentação exige **par**:

- `pixAddressKey` — chave de destino
- `pixAddressKeyType` — um de: `CPF`, `CNPJ`, `EMAIL`, `PHONE`, `EVP`

**Não** enviar `bankAccount` quando o fluxo é por chave Pix.

## 3.4 Respostas HTTP esperadas

| Código | Significado documentado |
|--------|-------------------------|
| 200 | Sucesso — transferência criada (`PENDING`, etc.) |

# 4. Requisitos da autorização crítica

## 4.1 Comportamento padrão (contas não-BaaS)

Documentação **Security** / **Segurança**:

> *"All transfers made on Asaas, except for BaaS accounts, must be approved via SMS token or APP token."*

Erro quando autorização crítica está ativa e nenhum substituto está configurado:

```json
{
  "errors": [{
    "code": "invalid_action",
    "description": "Your account has critical authorization enabled. To perform this action, enter the confirmation code."
  }]
}
```

Versão PT:

> *"Sua conta possui autorização crítica habilitada. Para efetuar esta ação informe o código de confirmação."*

## 4.2 Mecanismos substitutos oficiais (automação)

| Mecanismo | Efeito documentado |
|-----------|-------------------|
| **IP Whitelisting** | Apenas IPs fixados usam a API; token SMS pode ser removido com segurança |
| **Webhook de autenticação/autorização** | Substitui token manual — ver Seção 5 |
| **Desativação da autorização crítica** | Solicitar ao **account manager** (produção) ou **suporte** (sandbox) **após** habilitar um mecanismo acima |

## 4.3 `authToken` no body

- **Produção:** documentação de `POST /v3/transfers` **não** lista `authToken` como campo padrão do schema OpenAPI.
- **Sandbox:** testes de transferência com autorização crítica usam token de confirmação em fluxos de homologação (evidência interna F4.2G).
- **Conclusão:** omitir `authToken` em produção é correto **quando** IP whitelist ou webhook de validação substituem o SMS — não quando a conta ainda exige código na API sem mecanismo configurado.

## 4.4 Nota sobre Pix instantâneo (transfer-events)

Documentação **Events for transfers**:

> *"Non-scheduled Pix transfers are processed instantly. APP Token and SMS Token must be disabled."*

Isso reforça que, para Pix imediato automatizado, **tokens SMS/APP devem estar desabilitados** (via mecanismos substitutos + ação comercial/suporte).

# 5. Requisitos do webhook de validação

## 5.1 Dois webhooks distintos (regra P1.6)

| Webhook | URL Gol de Ouro | Momento | Payload |
|---------|-----------------|---------|---------|
| **Validação/autorização** | `/webhooks/asaas/transfer-validation` | **Antes** da execução Pix (~5 s após criação) | `type: "TRANSFER"` |
| **Financeiro** | `/webhooks/asaas` | **Após** mudança de estado | `event: "TRANSFER_DONE"`, etc. |

## 5.2 Fluxo oficial (ordem temporal)

Documentação **Mechanism for validating withdrawals via webhooks**:

1. Integrador faz `POST /v3/transfers` via API e armazena `transfer.id`.
2. **~5 segundos depois**, Asaas faz `POST` na URL configurada com payload da transferência.
3. Integrador responde **síncrono**: `{"status":"APPROVED"}` ou `{"status":"REFUSED","refuseReason":"..."}`.
4. Máximo **3 tentativas**; após 3 falhas → transferência **cancelada automaticamente**.
5. Se aprovada → Asaas executa a transferência → webhooks financeiros `TRANSFER_*`.

**Resposta à pergunta 6:** o endpoint `/webhooks/asaas/transfer-validation` **não precisa estar ativo antes** do `POST /v3/transfers` no sentido de pré-requisito de criação. Porém:

- Deve estar **configurado no painel Asaas** e **respondendo** quando a transferência for criada.
- No backend Gol de Ouro, `ASAAS_TRANSFER_AUTH_WEBHOOK_ENABLED=true` deve estar ativo **na janela** para o endpoint não retornar 404.

## 5.3 Habilitação na conta

| Fonte | Requisito |
|-------|-----------|
| PT — Withdrawals via webhooks | **Menu Usuário → Integrações → Mecanismos de Segurança** — URL, e-mail de erros, token (`asaas-access-token`) |
| EN — Transfers via webhooks | *"not enabled by default"* — *"contact technical support to enable"* — *"release subject to analysis"* — recomendado BaaS |
| Ambas | Após ativar: *"all transfers must be made exclusively via API"* |

## 5.4 Resposta e timeout

| Parâmetro | Valor documentado |
|-----------|-------------------|
| Resposta aprovação | `{"status":"APPROVED"}` |
| Resposta recusa | `{"status":"REFUSED","refuseReason":"..."}` |
| Timeout webhook geral (FAQ) | **10 segundos** |
| Retries auth webhook | **3** (depois cancela transferência) |
| Header token | `asaas-access-token` (opcional mas recomendado) |

# Apêndice A — Respostas detalhadas às 10 perguntas obrigatórias

### 1. Payload 100% compatível?

**Sim.** Todos os campos enviados constam na referência OpenAPI e guias Pix/TED. Nenhum campo proibido ou mutualmente exclusivo (`pixAddressKey` sem `bankAccount`).

### 2. Chave EMAIL aceita?

**Sim.** `TransferSaveRequestPixAddressKeyType` enum: `CPF`, `CNPJ`, `EMAIL`, `PHONE`, `EVP`. FAQ Pix inclui exemplo com e-mail.

### 3. `operationType=PIX` correto ou opcional?

**Opcional.** Se omitido com chave Pix, sistema define Pix automaticamente. Informar `PIX` explicitamente é **correto e claro** — não causa rejeição.

### 4. `externalReference` permitido?

**Sim.** Documentado como *"Identificador da transferência no seu sistema"*. FAQ recomenda uso para conciliação.

### 5. Exigência de authToken / código crítico?

**Condicional.** Contas não-BaaS: SMS/APP **ou** IP whitelist **ou** webhook de autenticação + desativação de autorização crítica. Sem isso → `invalid_action`. `authToken` no body **não** substitui configuração de conta em produção padrão.

### 6. Webhook auth ativo antes da criação?

**Não.** Ordem: `POST /v3/transfers` → armazenar id → ~5 s → Asaas POST no webhook → APPROVED/REFUSED. Pré-requisito: URL **configurada no painel Asaas** e endpoint **respondendo** quando a transferência existir.

### 7. Recurso específico na conta?

**Sim:**
- Saldo em `GET /finance/balance`
- Para automação: desativar autorização crítica **ou** habilitar mecanismos substitutos
- Webhook de validação: habilitação no painel (PT) ou suporte (EN/BaaS)
- Pix imediato: SMS/APP devem estar desabilitados (transfer-events)

### 8. `invalid_action` — causas documentadas

| Causa | Ligada? |
|-------|---------|
| Autorização crítica | ✅ **Sim — documentação explícita** |
| Conta sem permissão / mecanismo não liberado | ✅ Sim |
| Saldo | ❌ Improvável como `invalid_action` |
| Compliance | ⚠️ Possível — não mapeado na doc pública |
| Produção não habilitada | ❌ Seria 401/403 |
| Webhook auth ausente no POST | ❌ Não bloqueia criação; bloqueia execução pós-200 |

### 9. Penalização `/webhooks/asaas` afeta transferências?

**Na criação:** **Não.** `POST /v3/transfers` é chamada outbound autenticada por API Key.

**Na reconciliação:** **Sim, indiretamente.** Se o webhook financeiro estiver com fila pausada (15 falhas consecutivas, resposta ≠ HTTP 200), eventos `TRANSFER_DONE` / `TRANSFER_FAILED` não chegam — mas a transferência pode existir no Asaas.

Penalização é **por configuração de webhook** — webhook de auth e financeiro são filas separadas.

# Resumo Executivo

O `fly ssh` **não está bloqueado no lado Fly** — a aplicação está saudável, HTTPS responde, `fly doctor` passou em todos os checks e comandos remotos simples (`echo ok`, `hostname`, `printenv JWT_SECRET`) **executam com sucesso** no container.

O gargalo para P1.6F é **local ao Windows**, com **dois modos de falha distintos**:

| Modo | Sintoma | Duração típica | Impacto no homologador P1.6E |
|------|---------|----------------|------------------------------|
| **A — teardown Windows** | stdout válido + `Error: Identificador inválido.` + exit=1 | 3 s – 270 s | Secret **parseável**; homologador **passaria** se recebesse o stdout |
| **B — quoting `shell:true`** | `Error: host unavailable at JWT_SECRET: host was not found in DNS` + exit=1 | ~2,5–7 s | **Nenhum** secret no stdout → `auth-preflight` aborta |

**Causa raiz provável (evidência reproduzida nesta sessão):** em `scripts/lib/p16c-admin-auth.cjs`, `spawnSync` usa `shell: true` no Windows. Isso reescreve o argumento `-C printenv JWT_SECRET` sem aspas, o `flyctl` interpreta `JWT_SECRET` como **hostname interno** (não como parte do comando) e falha com erro DNS (**Modo B**). Com `shell: false` (mesmo padrão de `p16c-retry-homolog.cjs`, que já obteve o secret em P1.6C.RETRY), o comando retorna o secret em ~3 s e só então emite `Identificador inválido` (**Modo A**).

Nenhuma alteração operacional foi realizada nesta etapa.

### Comportamento por shell

| Shell | Comando | Resultado |
|-------|---------|-----------|
| **PowerShell** | `fly ssh … -C "echo ok"` | stdout `ok`, stderr `Identificador inválido`, exit **1**, ~245 s |
| **PowerShell** | `fly ssh … -C "hostname"` | stdout `080e207b071048`, stderr `Identificador inválido`, exit **1**, ~267 s |
| **PowerShell** | `fly ssh … -C "printenv JWT_SECRET"` (aspas) | stdout secret (linha base64), stderr `Identificador inválido`, exit **1**, ~104 s |
| **CMD** | `fly ssh … -C printenv JWT_SECRET` (sem aspas) | `host unavailable at JWT_SECRET`, exit **0** (CMD não propagou exit do fly) |
| **Node spawnSync** | `shell: false` | secret em ~3 s + `Identificador inválido`, exit **1** |
| **Node spawnSync** | `shell: true` (como P1.6E) | `host unavailable at JWT_SECRET`, exit **1**, ~2,7 s |

**Problema ocorre em PowerShell e CMD** — não é exclusivo de um shell. A variável decisiva é **como o argumento `-C` é passado ao processo** (`shell:true` vs `false` / aspas).

# Causa mais provável

**Cadeia causal em duas camadas:**

1. **Bloqueador imediato P1.6F (Modo B):** `resolveProductionJwtSecret()` em `p16c-admin-auth.cjs` usa `shell: process.platform === 'win32'`. No Windows, isso faz o cmd re-tokenizar `printenv JWT_SECRET` em dois tokens; o `flyctl` trata `JWT_SECRET` como alvo de host interno → `host unavailable at JWT_SECRET: host was not found in DNS`. O parser P1.6E rejeita corretamente essa linha → abort em ~6 s sem fallback `.env`.

2. **Comportamento estrutural Windows (Modo A):** mesmo com quoting correto (`shell:false` ou `-C "printenv JWT_SECRET"`), o `flyctl` retorna **exit=1** com `Error: Identificador inválido.` **depois** de imprimir o stdout útil. Isso é ruído de teardown local, **não** falha remota — já observado em P1.5Y, F3.3B e P1.6C.RETRY.

O homologador P1.6E está correto ao não usar fallback `.env`; falha porque a combinação `shell:true` + comando multi-palavra impede obter o secret, não porque o secret Fly esteja errado ou a máquina esteja down.

# Recomendações

> Todas abaixo são **sugestões para etapas futuras** — nenhuma foi aplicada em P1.6E.1 (READ-ONLY).

### Caminho imediato para P1.6F (sem alterar backend)

1. **Injeção manual (já suportada):**
   ```powershell
   $env:P16C_JWT_SECRET = "<valor obtido via fly ssh com aspas ou dashboard>"
   $env:P16C_PHASE = "auth-preflight"
   node scripts/p16c-homolog-final.cjs
   ```
   Obter o valor com:
   ```powershell
   fly ssh console -q -a goldeouro-backend-v2 -C "printenv JWT_SECRET"
   ```
   (ignorar exit 1 se a linha do secret aparecer no stdout)

### Correção cirúrgica homologador (P1.6E.2 — quando autorizado)

1. Trocar `shell: true` → **`shell: false`** em `spawnSync` do `p16c-admin-auth.cjs` (alinhar com `p16c-retry-homolog.cjs`).
2. Opcional: usar `-C "printenv JWT_SECRET"` como string única (padrão `f2-4e-2a-approve-send.mjs`).
3. **Não** abortar só por `r.status !== 0` se `parseFlyJwtSecretOutput()` encontrou secret válido (tratar `Identificador inválido` como ruído Windows).

### Mitigações operacionais Fly (se latência persistir)

1. `fly wireguard websockets enable` (documentado na comunidade Fly para redes restritivas).
2. Executar homologação de uma sessão WSL/Linux/GitHub Actions onde SSH já foi estável.
3. Manter `flyctl` atualizado; monitorar [status.flyio.net](https://status.flyio.net) para incidentes `fly ssh console`.

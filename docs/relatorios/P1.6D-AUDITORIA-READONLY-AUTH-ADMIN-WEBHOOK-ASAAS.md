## Token administrativo

| Aspecto | Comportamento do script |
|---------|-------------------------|
| Formato | **JWT** (`jsonwebtoken.sign`) |
| Header | `Authorization: Bearer <jwt>` |
| Payload | `{ userId, email }` do actor |
| Actor | Prioriza `TECH_USER` (`85872488-…`); fallback primeiro `tipo=admin` no DB |
| Segredo | `getFlyJwtSecret()` → `fly ssh … printenv JWT_SECRET` **ou** `process.env.JWT_SECRET` |
| Algoritmo | Default da lib **`HS256`** (não especificado no `sign`) |
| Expiração | `15m` |
| Bearer fixo | **Não** |

# Auditoria do webhook Asaas

## Rota `POST /webhooks/asaas`

```javascript
// server-fly.js L3451–3459
if (!isPaymentWebhookEngineEnabled()) → 404 payment_webhook_engine_disabled
if (!isAsaasWebhookRouteAllowed())     → 403 asaas_webhook_blocked
// só então processPaymentWebhookCompat → validação token/assinatura Asaas
```

## `isAsaasWebhookRouteAllowed()` (processPaymentWebhook.js L29–34)

Retorna `false` quando:

1. `ASAAS_WEBHOOK_ENABLED` ≠ `true`, **ou**
2. Sandbox webhook não permitido **e** `ASAAS_PRODUCTION_ENABLED` ≠ `true`, **ou**
3. `NODE_ENV=production` **e** `ASAAS_PRODUCTION_ENABLED` ≠ `true`

## Flags relevantes (estado nominal pós-janela, per P1.6C.FINAL-PRECHECK)

| Flag | Estado típico pós-homolog | Efeito no webhook financeiro |
|------|---------------------------|------------------------------|
| `PAYMENT_WEBHOOK_ENGINE_ENABLED` | **true** | Engine ativa → passa 1º check |
| `ASAAS_WEBHOOK_ENABLED` | **true** | OK para Asaas |
| `ASAAS_PRODUCTION_ENABLED` | **false** | **Bloqueia** em produção → **403** |
| `ASAAS_PIX_OUT_PRODUCTION_ENABLED` | **false** | **Não** consultado em `isAsaasWebhookRouteAllowed` |
| `PAYMENT_ENGINE_PIXOUT_ENABLED` | **false** | **Não** consultado na rota webhook |
| `ASAAS_PIX_IN_ENABLED` | **true** | **Não** consultado na rota webhook |

## Respostas às perguntas-chave

| Pergunta | Resposta |
|----------|----------|
| Por que HTTP 403? | `ASAAS_PRODUCTION_ENABLED=false` em runtime `NODE_ENV=production` |
| Depende de PIX OUT? | **Não diretamente** — gate é `ASAAS_PRODUCTION_ENABLED` (compartilhado) |
| Depende só de produção? | Sim, na prática: produção + gate OFF = 403 |
| Depende de assinatura/token Asaas? | **Não neste 403** — bloqueio ocorre **antes** do validator |
| Deveria aceitar PIX IN com PIX OUT fechado? | **Operacionalmente desejável**, mas **desenho atual não permite** — um único gate bloqueia toda a rota |
| PIX IN e PIX OUT desacoplados? | **Parcialmente** — HTTP PIX OUT usa gates próprios; **webhook financeiro `/webhooks/asaas` é monolítico** |
| Risco de penalização Asaas? | **Sim** — eventos reais de produção recebem 403 enquanto gate OFF; Asaas reitera e pode penalizar (documentado em P1.5W) |

## Rota separada de autorização (contexto P1.6)

`POST /webhooks/asaas/transfer-validation` usa gate **`ASAAS_TRANSFER_AUTH_WEBHOOK_ENABLED`** — **independente** do webhook financeiro. O script de homolog, ao fechar gates, também seta `ASAAS_TRANSFER_AUTH_WEBHOOK_ENABLED=false` (L119).

# Diagnóstico do erro "invalid signature"

## Mecanismo

1. Script assina JWT com segredo obtido por `getFlyJwtSecret()`.
2. Backend em Fly verifica com `jwt.verify(token, process.env.JWT_SECRET)` (segredo **do container Fly**).
3. `jsonwebtoken` emite `JsonWebTokenError: invalid signature` quando segredos diferem.

## Causa raiz

**Mismatch `JWT_SECRET` local vs produção.**

| Origem | Evidência |
|--------|-----------|
| Fly produção | Digest `JWT_SECRET` = `2c6d94ec…` (lista secrets P1.6C / prep) |
| `.env` local | Contém `JWT_SECRET` próprio (carregado por `dotenv`) |
| Fallback L133 | Se `fly ssh` falha ou stdout não é parseado corretamente no Windows → usa `.env` local |
| Log operador | `[AUTH] Token inválido: invalid signature` — típico de assinatura com segredo errado |

## Por que não é `ADMIN_TOKEN`

A rota **não** compara com `ADMIN_TOKEN`. Token fixo inválido como JWT também geraria `invalid signature`, mas o script P1.6C **gera JWT**, não usa `ADMIN_TOKEN`.

## Bloqueador adicional (pós-correção de assinatura)

Se JWT passar com conta técnica não-admin → **403** *“Acesso restrito a administradores”* (`requireAdministradorDb` L2080–2081) — distinto do erro observado.

# Diagnóstico do HTTP 403

## Mecanismo

```
Asaas → POST /webhooks/asaas
     → isPaymentWebhookEngineEnabled() = true
     → isAsaasWebhookRouteAllowed() = false  (ASAAS_PRODUCTION_ENABLED=false)
     → HTTP 403 { error: "asaas_webhook_blocked", ... }
```

## Causa raiz

**Comportamento esperado e documentado** com gates financeiros fechados após homologação (P1.2, P1.5W, P1.6C.FINAL-PRECHECK).

## Relação com a janela P1.6C.RETRY

Durante a janela (~100s), o script seta `ASAAS_PRODUCTION_ENABLED=true` — webhooks **seriam aceitos** nesse intervalo. Após `closeGates()`, `ASAAS_PRODUCTION_ENABLED=false` → **403 retorna** para qualquer evento Asaas em produção, incluindo:

- `PAYMENT_RECEIVED` (PIX IN)
- `TRANSFER_DONE` (PIX OUT)

Isso explica penalização no painel Asaas **independente** de PIX OUT estar fechado.

# Recomendações futuras

> **Somente orientação — fora do escopo desta auditoria (sem implementar).**

1. **Auth homolog:** obter JWT de admin via `POST /api/auth/login` (email/senha admin) **ou** garantir `JWT_SECRET` idêntico ao Fly **sem** depender de `fly ssh` no Windows.
2. **Actor:** assinar JWT com usuário `tipo=admin` confirmado no DB, não conta técnica de validação.
3. **Webhook 403:** alinhar expectativa operacional — enquanto `ASAAS_PRODUCTION_ENABLED=false`, **não** manter webhook de produção ativo no painel Asaas; ou abrir gate temporário para reprocessamento (runbook P1.5W).
4. **Desacoplamento:** avaliar gate separado PIX IN webhook vs PIX OUT (arquitetura futura) — hoje um gate serve ambos.
5. **Pré-retry:** validar assinatura localmente (`jwt.verify` com secret Fly) antes de abrir janela gates.
6. **Artefatos:** preservar `tmp/p16c-homolog-state.json` fora do gitignore em relatórios ou copiar para `docs/relatorios/` após execução.

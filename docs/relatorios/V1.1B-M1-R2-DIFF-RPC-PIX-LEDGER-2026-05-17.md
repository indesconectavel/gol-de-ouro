## 2. Baseline produção (inferida pré-R2)

Características que **devem ser preservadas** (requisitos da missão):

| Componente | Comportamento baseline |
|------------|----------------------|
| Advisory lock | `pg_advisory_xact_lock(hashtextextended('claim_pix_deposit:' \|\| payment_id, 0))` |
| Claim | `UPDATE pagamentos_pix` pending → approved (condicional) |
| Ledger | `INSERT` tipo `deposito` |
| Idempotência ledger | Detecção por `(correlation_id, tipo)` — cobre 8 legados `referencia = payment_id` |
| Ramo pending | claim + ledger + saldo (atómico) |

**Self-heal inferido (bug alvo):** quando `status = approved` e **sem** linha `deposito` para o MP id, a RPC em produção **também** faz `UPDATE usuarios.saldo += credit` (ou equivalente), devolvendo `credited: true`. Isso:

- **Correcto** para os 4 suspeitos (saldo 0, nunca creditados) — mas esses devem ficar **fora** do invoke automático (runbook).
- **Incorrecto** para ~30 approved já creditados → **double credit** no primeiro reconcile/webhook pós-deploy.

O rascunho M1 em `database/claim_and_credit_approved_pix_deposit.sql` **já** implementa self-heal só-ledger (`ledger_backfill`, `credited: false`) — R2 **não** propõe reaplicar esse ficheiro inteiro; propõe **apenas** alinhar o ramo self-heal da RPC **real** a esse comportamento.

## 3. Diff cirúrgico (único hunk funcional)

### 3.1 Remover (baseline inferida com bug)

```sql
-- [SELF-HEAL] IF v_status_norm = 'approved' THEN
--   ... INSERT ledger ...
--   SELECT saldo ... FOR UPDATE;
--   UPDATE usuarios SET saldo = saldo + v_credit;   -- ← REMOVER
--   RETURN ... 'credited', true ...;                 -- ← REMOVER
```

### 3.2 Inserir (R2)

```sql
IF v_status_norm = 'approved' THEN
  INSERT INTO ledger_financeiro (...)  -- deposito, referencia = pagamentos_pix.id::text
  ON CONFLICT (correlation_id, tipo, referencia) DO NOTHING
  ...
  RETURN jsonb_build_object(
    'ok', true,
    'credited', false,
    'idempotent', true,
    'reason', 'ledger_backfill',
  ...
  );
  -- SEM UPDATE em usuarios.saldo
END IF;
```

### 3.3 Inalterado

- Advisory lock (início da função)
- Lookup `pagamentos_pix` + `FOR UPDATE`
- Check ledger `(correlation_id, tipo)`
- Ramo `pending`: claim → ledger → saldo
- Ramo concorrência (`v_rows_claimed = 0` + já approved)
- `EXCEPTION` handler

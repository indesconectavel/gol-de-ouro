# Critério GOLD

**Após P1.8, é possível perder definitivamente um PIX OUT por falha de webhook?**

## NÃO.

O Recovery Job consulta periodicamente `GET /v3/transfers/{id}` para saques em `aguardando_confirmacao` e reconcilia estados terminais (`DONE`, `FAILED`, etc.) via o mesmo pipeline idempotente do webhook. Mesmo com fila Asaas pausada e sem eventos `TRANSFER_*` inscritos, o saque P1.7C (`f3723ce8…` / transfer `5d1355b9…`) será reconciliado automaticamente na próxima execução do job com API key válida.

# ETAPA C — Implementação

## O que já existia (mantido intacto)

Handlers `TRANSFER_DONE` / `TRANSFER_FAILED` em `processAsaasTransferWebhook.js` — **não reimplementados**.

## O que foi adicionado em P1.8

### 1. Logs estruturados

```text
[ASAAS][TRANSFER_DONE]  { saqueId, transferId, externalReference, source, idempotent, financialEffect }
[ASAAS][TRANSFER_FAILED] { saqueId, transferId, transferStatus, source, rollbackApplied, idempotent }
```

`source` = `webhook` | `recovery_job`

### 2. Gate HTTP read-only produção

`isAsaasPayoutReconcileReadEnabled()` em `asaas-config.js`  
`httpGate: 'payoutReconcile'` em `asaas-http-client.js`  
Permite `GET /transfers/{id}` em produção com `ASAAS_ENABLED` + API key, **sem** gates de escrita PIX OUT.

### 3. Recovery Job

**Arquivo:** `src/finance/reconciliation/asaasPayoutRecovery.js`

Fluxo:

```text
SELECT saques
  WHERE status='aguardando_confirmacao'
    AND asaas_transfer_id IS NOT NULL
        │
        ▼
GET /v3/transfers/{transfer_id}  (payoutReconcile gate)
        │
        ├─ DONE      → TRANSFER_DONE sintético → reconciliar
        ├─ FAILED    → TRANSFER_FAILED sintético → rollback
        ├─ CANCELLED → TRANSFER_CANCELLED sintético → rollback
        └─ PENDING   → não alterar
```

Validações: `transfer_id` match, `externalReference` match, idempotência via handler existente.

### 4. Scheduler

**Arquivo:** `server-fly.js` — espelha padrão `reconcilePendingPayments` (MP)

| Variável | Default | Descrição |
|----------|---------|-----------|
| `ASAAS_PAYOUT_RECOVERY_ENABLED` | true (opt-out: `false`) | Habilita job |
| `ASAAS_PAYOUT_RECOVERY_INTERVAL_MS` | 120000 | Intervalo mínimo 60s |
| `ASAAS_PAYOUT_RECOVERY_LIMIT` | 20 | Saques por ciclo |
| `ASAAS_PAYOUT_RECOVERY_MIN_AGE_MIN` | 1 | Idade mínima antes de consultar |

# ETAPA D — Recovery Job (detalhes)

## Idempotência

| Guard | Mecanismo |
|-------|-----------|
| Saque terminal | Skip se `status ≠ aguardando_confirmacao` |
| Replay webhook | `isIdempotentReplay()` |
| Ledger duplicado | `findExistingTerminalLedger()` → `payout_confirmado` / `falha_payout` |
| Rollback duplicado | `rollbackWithdraw()` verifica ledger `rollback` existente |
| Reconciliação dupla | Segunda execução retorna `idempotent` / `ledgerIdempotent` |

## Comportamento por status Asaas

| Status API | Ação Recovery |
|------------|---------------|
| `DONE` | `status=processado`, `payout_confirmado`, `[ASAAS][TRANSFER_DONE]` |
| `FAILED` / `CANCELLED` / `BLOCKED` | `falha_payout`, `rollbackWithdraw`, `[ASAAS][TRANSFER_FAILED]` |
| `PENDING` / `BANK_PROCESSING` | **Não altera** |

# Riscos Remanescentes

| Risco | Mitigação | Severidade |
|-------|-----------|------------|
| API Asaas indisponível prolongadamente | Job retenta no próximo ciclo; estado permanece `aguardando_confirmacao` | Média |
| `ASAAS_API_KEY` ausente/inválida | Gate `payoutReconcile` bloqueia; logs `ASAAS_PAYOUT_RECONCILE_READ_DISABLED` | Alta (ops) |
| Webhook ainda sem `TRANSFER_*` | Recovery compensa; latência até próximo ciclo (~2 min default) | Baixa |
| Fila Asaas pausada | Recovery independente de webhook | Resolvido P1.8 |
| Limite `ASAAS_PAYOUT_RECOVERY_LIMIT` | Backlog grande pode demorar múltiplos ciclos | Baixa |

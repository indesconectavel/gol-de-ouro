### Por que ficou `PENDING` + `authorized: false`?

A conta Sandbox possui **autorização crítica habilitada** (padrão). A transferência foi criada via API **sem** token crítico na requisição inicial (F4.2G.1). O campo `authorized` indica explicitamente: *"false when awaiting authorization via SMS Token"*.

### Como autorizar no Sandbox (métodos oficiais)

| Método | Documentado | Automatizável via API nesta fatia |
|--------|-----------|----------------------------------|
| Token crítico `000000` na **criação** (`POST /transfers`) | Sim ([How to Test Critical Actions](https://docs.asaas.com/docs/how-to-test-critical-actions)) | Campo do body **não** consta na referência OpenAPI — não implementado |
| Aprovação manual no **painel** Sandbox | Sim | Não (operador) |
| Webhook validação → `APPROVED` | Sim | Requer URL pública (não montada) |
| Whitelist IP + desativar evento crítico | Sim | Configuração de conta, não código |
| Contato Integration Success — desabilitar token Sandbox | Sim | Não |

**Não burlar 2FA.** **Não inventar endpoint de autorização.**

## 11. Encerramento F4.2G.3 (2026-06-27)

A fatia seguinte executou o procedimento oficial de token crítico `000000` na criação (`POST /transfers`).

| Item | Resultado F4.2G.3 |
|------|-------------------|
| Modo testado (sucesso HTTP 200) | `body.authToken` com valor `000000` |
| `authorized` após criação | `false` (inalterado) |
| Status após 15 consultas | `PENDING` (inalterado) |
| `DONE` alcançado? | **Não** |
| Classificação | `VALIDATED_WITH_OPERATIONAL_LIMITATION` |

**Conclusão encadeada:** a hipótese F4.2G.2 (autorização crítica bloqueia evolução) foi **confirmada em live test**. O token documentado para Sandbox, enviado no body, **não substitui** a autorização crítica nesta conta/configuração. A forma exata do parâmetro não consta na OpenAPI.

Relatório: [F4.2G.3-TRANSFER-DONE-VALIDATION.md](./F4.2G.3-TRANSFER-DONE-VALIDATION.md)

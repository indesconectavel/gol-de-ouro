## Veredito da Auditoria

# SUPORTE OFICIAL CONFIRMADO

O Asaas oferece mecanismo **oficial, documentado e suportado** para substituir a autorização via Token SMS/APP por validação server-to-server via webhook. A implementação P1.6 é **elegível para prosseguir**, condicionada à **ativação na conta Asaas** (não habilitado por padrão) e à **desativação da autorização crítica** após habilitar um mecanismo de segurança adicional.

## 2. Respostas Obrigatórias (16 Questões)

### 1. Existe mecanismo oficial de autorização automática?

**Sim.** O Asaas documenta o **"Transfer Authorization Webhook"** (também chamado *Mechanism for validating transfers/withdrawals via webhooks*). Após criar a transferência via API, o Asaas envia um `POST` ao endpoint configurado; a aplicação responde com `{"status": "APPROVED"}` ou `{"status": "REFUSED", "refuseReason": "..."}` para autorizar ou negar a execução — **sem intervenção humana**.

### 2. É suportado para contas Asaas comuns?

**Sim, com ressalva.** Contas comuns (não-BaaS) exigem por padrão aprovação via SMS/APP para transferências. A documentação de **Security** afirma explicitamente que é possível **substituir o token** por:
- **IP Whitelisting**, ou
- **Webhook de autenticação/autorização**

Após habilitar um desses mecanismos, solicita-se ao gerente de conta ou suporte a **desativação da autorização crítica** em produção.

### 3. É suportado apenas para contas específicas?

**Parcialmente.** A página *Mechanism for validating transfers via webhooks* indica que o recurso é **"mainly recommended for BaaS account uses"** e que a **liberação está sujeita a análise**. Contudo, a página *Mechanism for validating withdrawals via webhooks* documenta ativação via **Menu do Usuário → Integrações → Mecanismos de Segurança** para qualquer conta, com replicação automática para subcontas.

**Conclusão:** suportado oficialmente; contas BaaS têm fluxo nativo sem SMS; contas comuns precisam de habilitação + desativação da autorização crítica.

### 4. Exige contratação?

**Não há menção a plano ou produto adicional pago.** O mecanismo faz parte dos **Mecanismos de Segurança Adicionais** documentados gratuitamente na API.

### 5. Exige aprovação comercial?

**Sim, potencialmente.** A versão EN do doc de transferências declara: *"The release of use is subject to analysis"* e *"you will need to contact technical support to enable it"*. A versão PT-BR (saques) permite autoatendimento via painel em **Integrações → Mecanismos de Segurança**.

**Ação operacional:** configurar no painel; se indisponível, abrir ticket ao suporte informando URL + authToken.

### 6. Exige liberação manual?

**Sim — uma vez.** O mecanismo **não vem habilitado por padrão**. Requer:
1. Configurar URL do webhook + e-mail de erros + token (opcional mas recomendado) no painel Asaas, **ou**
2. Contatar suporte técnico informando URL e authToken (versão EN do doc de transferências).

Adicionalmente, para eliminar SMS em contas comuns: solicitar ao **account manager** desativação da autorização crítica após habilitar IP whitelist ou webhook de autorização.

### 7. Existe endpoint oficial?

**Não é um endpoint que nós chamamos no Asaas.** O fluxo é **inbound**: o Asaas faz `POST` na **URL que configuramos** (~5 segundos após `POST /v3/transfers`). Não existe `POST /transfers/{id}/authorize` na referência API pública (confirmado em F4.2G.2).

### 8. Existe webhook oficial?

**Sim.** Webhook dedicado de **validação/autorização**, distinto dos webhooks financeiros `TRANSFER_*` (`TRANSFER_DONE`, etc.).

| Tipo | Propósito | Formato |
|------|-----------|---------|
| **Authorization Webhook** | Aprovar/recusar transferência antes da execução | `{"type":"TRANSFER","transfer":{...}}` |
| **Financial Webhook** | Notificar mudança de estado pós-execução | `{"id":"evt_...","event":"TRANSFER_DONE","transfer":{...}}` |

### 9. Existe payload oficial?

**Sim.** Payload documentado:

```json
{
  "type": "TRANSFER",
  "transfer": {
    "object": "transfer",
    "id": "0bed986c-737d-49bf-a1cc-beca916797c4",
    "status": "PENDING",
    "value": 22,
    "netValue": 22,
    "operationType": "PIX",
    "externalReference": null,
    "pixAddressKey": null,
    "...": "..."
  }
}
```

Resposta de aprovação:

```json
{ "status": "APPROVED" }
```

Resposta de recusa:

```json
{ "status": "REFUSED", "refuseReason": "Motivo da recusa" }
```

### 10. Existe timeout?

**Sim.**
- **Webhook de autorização:** não documenta timeout explícito na página do mecanismo; a FAQ geral de webhooks indica **10 segundos** de espera para resposta.
- **Implicação:** responder de forma **síncrona e rápida** (< 10s) com `APPROVED`/`REFUSED`.

### 11. Existe retry?

**Sim — específico para autorização.**
- Máximo **3 tentativas** de entrega do webhook de autorização.
- Após a **3ª falha** (sem status válido ou erro HTTP), a transferência é **cancelada automaticamente**.

Isso difere dos webhooks financeiros (até 15 falhas consecutivas antes de pausar a fila).

### 12. Existe assinatura?

**Não há HMAC/Bearer padrão.** A FAQ confirma: *"Asaas Webhooks do not use standard Bearer Token authentication"*. A validação recomendada é via header customizado **`asaas-access-token`** configurado no painel.

### 13. Existe token?

**Sim — opcional mas recomendado.**
- Configurado no painel Asaas (Mecanismos de Segurança).
- Enviado no header **`asaas-access-token`** em cada `POST` de autorização.
- Deve ser validado com comparação segura (timing-safe) no nosso endpoint.

### 14. Existe whitelist?

**Sim — mecanismo complementar.**
- **IP Whitelisting:** apenas IPs autorizados podem usar a API key.
- **IPs oficiais de webhook (produção):** `52.67.12.206`, `18.230.8.159`, `54.94.136.112`, `54.94.183.101`.
- Sandbox pode usar IPs adicionais (verificar em Security → Events no Cloudflare).

Recomendação Asaas: usar **pelo menos um** mecanismo (IP whitelist **e/ou** webhook de autorização); idealmente ambos.

### 15. Existe limitação?

**Sim — limitações documentadas:**

| Limitação | Detalhe |
|-----------|---------|
| Não habilitado por padrão | Requer configuração manual |
| Liberação sujeita a análise | Versão EN — contatar suporte |
| Todas as transferências via API | Após ativar, transferências devem ser exclusivamente via API |
| Escopo total | Todas as transferências/saques API passam pelo mecanismo |
| Subcontas | Configuração replicada automaticamente |
| 3 falhas = cancelamento | Janela curta para responder |
| SMS padrão em contas comuns | Só eliminado após habilitar mecanismo + desativar autorização crítica |
| Distinto de webhooks financeiros | URL separada recomendada |

### 16. Existe fluxo recomendado?

**Sim.** Fluxo oficial:

```
1. POST /v3/transfers (API) → armazenar transfer.id + dados
2. ~5s depois → Asaas POST no webhook de autorização
3. Validar payload vs. base de dados
4. Responder APPROVED ou REFUSED (síncrono)
5. Se APPROVED → Asaas executa transferência
6. Webhooks financeiros TRANSFER_* notificam conclusão
```

**Boas práticas adicionais (docs):**
- Validar `asaas-access-token`
- Allowlist de IPs Asaas no firewall/WAF
- Idempotência (mesmo transfer.id pode ser reenviado)
- Não misturar com processamento assíncrono de webhooks financeiros
- Monitorar Webhook Logs no painel Asaas

## 3. Relação com o Bloqueio `invalid_action` (P1.5Y)

A documentação **Security** explica o erro observado na homologação P1.5:

```json
{
  "errors": [{
    "code": "invalid_action",
    "description": "Your account has critical authorization enabled. To perform this action, enter the confirmation code."
  }]
}
```

**Causa:** autorização crítica (SMS/APP) habilitada.  
**Solução oficial:** habilitar IP whitelist **ou** webhook de autorização, depois solicitar desativação da autorização crítica ao account manager (produção) ou suporte (sandbox).

**Evidência P1.5CERT:** após resolução parcial, `POST /v3/transfers` retornou HTTP 200, mas transferência permaneceu `PENDING` aguardando Token SMS — exatamente o cenário que o webhook de autorização elimina.

## 4. Distinção Crítica: Dois Webhooks Diferentes

| Dimensão | Authorization Webhook | Financial Webhook (`/webhooks/asaas`) |
|----------|----------------------|---------------------------------------|
| URL | `/webhooks/asaas/transfer-validation` (proposta P1.6) | `/webhooks/asaas` (existente P1.5G) |
| Payload | `type: "TRANSFER"` | `event: "TRANSFER_DONE"` |
| Resposta | `{"status":"APPROVED"}` | HTTP 200 + processamento assíncrono |
| Momento | Antes da execução | Após mudança de estado |
| Retry | Máx. 3, depois cancela | Até 15, depois pausa fila |
| Efeito financeiro | Nenhum (só autoriza) | Reconcilia wallet/ledger |

**Regra P1.6:** nunca reutilizar `/webhooks/asaas` para autorização.

## 6. Pré-requisitos Operacionais (Antes de Produção)

1. Configurar no painel Asaas: URL `https://<backend>/webhooks/asaas/transfer-validation`
2. Definir `authToken` e replicar em `ASAAS_TRANSFER_AUTH_TOKEN` (Fly secrets)
3. Habilitar `ASAAS_TRANSFER_AUTH_ENABLED=true` no backend
4. Allowlist IPs Asaas no WAF/Fly (opcional mas recomendado)
5. Solicitar ao account manager desativação da **autorização crítica** em produção
6. Validar em Sandbox antes de janela operacional

|------|---------|--------|
| `ASAAS_PIX_IN_ENABLED` | `false` | Gate feature PIX IN |
| `ALLOW_ASAAS_SANDBOX_PIX_IN` | `0` | Gate script/cobrança sandbox |
| `ASAAS_ENABLED` | `false` | Gate geral |

## Isolamento de Ambiente (F4.2D.1)

### Problema encontrado

Após `test-asaas-pix-in.mjs`, as flags `ASAAS_PIX_IN_ENABLED=true` e `ALLOW_ASAAS_SANDBOX_PIX_IN=1` permaneceram no processo shell, fazendo `verify-asaas-provider.mjs` falhar (`createPixDeposit` deixou de retornar `ASAAS_PAYMENT_NOT_IMPLEMENTED`).

### Causa

Scripts liam `process.env` sem resetar flags controladas e sem restaurar o ambiente ao finalizar.

### Solução adotada

Helper compartilhado `scripts/helpers/asaas-test-env.mjs` com:

- `snapshotEnvironment()` / `restoreEnvironment()`
- `resetAsaasEnvironment()` — baseline seguro determinístico
- `bootstrapAsaasTestEnv()` — dotenv + reset + preservação de credenciais
- `installExitRestore()` — restauração no exit
- `clearAsaasModuleCache()` — invalidação de require cache
- Perfis `applyAuthLiveProfile()` / `applyPixInLiveProfile()`

Scripts de teste HTTP real exigem **`ASAAS_TEST_LIVE=1`** (modo live explícito). Sem isso → `SKIP` determinístico.

Scripts de verificação aplicam baseline no início e restauram snapshot no `finally`.

### Estratégia anti-regressão

1. Toda flag Asaas declarada explicitamente por script.
2. Verificações nunca dependem de flags PIX/webhook deixadas por outros testes.
3. Regressão validada em ordens A e B com shell pré-contaminado — todos passaram.

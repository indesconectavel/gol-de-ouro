## 2. Problema encontrado

```text
[AUTH] Token inválido: invalid signature
```

Durante P1.6C.RETRY (`tmp/p16c-homolog-state.json`, `2026-06-30T05:51Z`):

1. Janela PIX OUT aberta corretamente.
2. `fly ssh … printenv JWT_SECRET` falhou (`host unavailable at JWT_SECRET`).
3. Script antigo usou segredo local incorreto e assinou JWT como conta técnica.
4. `POST approve-and-send` → HTTP **403** `Token inválido`.
5. Backend Fly registrou: `❌ [AUTH] Token inválido: invalid signature`.

---

## 3. Causa raiz

| # | Causa |
|---|--------|
| 1 | **Fallback inseguro:** `getFlyJwtSecret()` retornava `process.env.JWT_SECRET` do `.env` quando `fly ssh` falhava — digest Fly `2c6d94ec…` ≠ secret local. |
| 2 | **Parser frágil:** linhas de erro do `fly ssh` podiam ser tratadas como candidatas a secret. |
| 3 | **Actor incorreto:** JWT assinado para conta técnica; rota exige `requireAdministradorDb` (`tipo=admin`). |
| 4 | **Auth tardia:** validação ocorria só no POST financeiro, após abrir gates. |

## 5. Correções implementadas

### 5.1 Eliminação do fallback inseguro

- Removido uso de `process.env.JWT_SECRET` (dotenv) como fallback silencioso.
- Fontes permitidas:
  1. `P16C_JWT_SECRET` — override explícito do operador (valor Fly, nunca logado).
  2. `fly ssh console -q -a goldeouro-backend-v2 -C printenv JWT_SECRET`
- Se ambas falharem → **throw** com mensagem clara (sem prosseguir para `approve-and-send`).

### 5.2 Parser Windows-safe

`parseFlyJwtSecretOutput()`:

- Usa **primeira** linha válida (não a última).
- Rejeita `Connecting…`, `Error:`, `host unavailable`, `JWT_SECRET:` em mensagens de erro.
- Exige charset compatível com secret base64/ascii (`^[A-Za-z0-9+/_=\-]+$`, len ≥ 32).

### 5.3 Identidade administrativa

- `resolveAdminActor()` busca `tipo=admin` no Supabase.
- Override opcional: `P16C_ADMIN_USER_ID` (deve ser admin).
- **Removido** uso prioritário da conta técnica `85872488-…`.

### 5.4 JWT compatível com backend

- `signAdminJwt()` — `HS256`, `expiresIn: 15m`, payload `{ userId, email }` (igual padrão login).
- `verifyAdminJwtLocally()` — valida assinatura e claims antes de qualquer POST.

### 5.5 Pré-validação remota (read-only)

- `probeAdminAuth()` → `GET /api/admin/withdraw/list?limit=1`
- Confirma JWT + middleware admin **sem** operação financeira.
- Detecta explicitamente `invalid signature` vs falta de privilégio admin.

### 5.6 Logs objetivos (sem secrets)

Exemplos emitidos:

```text
Obtendo JWT_SECRET de produção…
JWT_SECRET obtido (valor omitido).
Perfil administrativo confirmado: userId=xxxxxxxx…xxxx tipo=admin
JWT assinado (token omitido).
Validando claims localmente…
Pré-validação remota: GET /api/admin/withdraw/list (read-only)…
Pré-validação administrativa OK (HTTP 200).
```

Apenas `sha256_prefix` (16 chars) do secret quando origem é conhecida.

### 5.7 Nova fase `P16C_PHASE=auth-preflight`

Valida autenticação **sem**:

- abrir gates;
- `POST approve-and-send`;
- PIX.

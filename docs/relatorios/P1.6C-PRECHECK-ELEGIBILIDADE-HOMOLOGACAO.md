# FASE 2 — Auditoria da taxa

## 2.1 Fluxo de cálculo

```
Valor solicitado (bruto)
        ↓
Taxa = parseFloat(process.env.PAGAMENTO_TAXA_SAQUE || '2.00')
        ↓
Valor líquido = valor solicitado − taxa
        ↓
net_amount persistido em saques
        ↓
PIX OUT Asaas usa net_amount como value em POST /v3/transfers
```

| Etapa | Implementação |
|-------|---------------|
| Taxa | `server-fly.js:1796` — `PAGAMENTO_TAXA_SAQUE`, default **R$ 2,00** |
| Líquido | `server-fly.js:1797` — `valorLiquido = requestedAmount - taxa` |
| Bloqueio líquido ≤ 0 | `server-fly.js:1798–1802` |
| Persistência | `server-fly.js:1839–1840` — `fee`, `net_amount` |
| Envio Asaas | `processPendingWithdrawals.js:885–890` — `netAmount` do saque |

## 2.2 Exemplo padrão de homologação

| Campo | Valor |
|-------|-------|
| Valor solicitado | R$ 10,00 |
| Taxa (default) | R$ 2,00 |
| Valor líquido (`net_amount`) | **R$ 8,00** |
| Valor debitado do wallet | R$ 10,00 |
| Valor enviado ao Asaas (`POST /transfers`) | **R$ 8,00** |

## 2.3 Perguntas específicas

| Pergunta | Resposta |
|----------|----------|
| Existe valor mínimo após taxa? | **Implícito:** `valorLiquido > 0` → bruto > taxa (R$ 2). Na prática o mínimo bruto R$ 10 domina |
| Existe arredondamento? | **Não** — `parseFloat` direto, sem `Math.round`/`toFixed` no cálculo |
| Bloqueio se líquido abaixo de X? | **Sim** — bloqueio genérico `valorLiquido <= 0` (`server-fly.js:1798`) |
| Taxa Asaas adicional (`transferFee`)? | Capturada em resposta (`asaas-http-client.js:555`) — não deduzida no cálculo interno; pode consumir saldo Asaas além do `value` |

## 2.4 Limites Asaas (camada provider — distinta do mínimo de negócio)

| Ambiente | Mín | Máx | Arquivo | Linha |
|----------|-----|-----|---------|-------|
| Produção PIX OUT | R$ 1 | R$ 1.000 | `asaas-http-client.js` | 28–29, 607–612 |
| Sandbox PIX OUT | R$ 5 | R$ 10 | `asaas-http-client.js` | 25–26 |

O provider Asaas **permitiria** transferência de R$ 1 em produção, mas o Gol de Ouro **bloqueia** saques abaixo de R$ 10 antes de chegar ao provider.

# FASE 3 — Auditoria de elegibilidade (saldo ≈ R$ 5,19)

## 3.1 Duas interpretações

### A) Criar saque técnico via API (débito wallet)

| Critério | Avaliação |
|----------|-----------|
| Mínimo R$ 10 | Conta técnica pode solicitar R$ 10 se wallet ≥ R$ 10 |
| Wallet técnica | ~R$ 166.555 — **suficiente** |
| Saldo Asaas R$ 5,19 | **Irrelevante** na criação do saque |
| Saques pendentes | **0** — criação seria permitida |

**Resposta parcial:** é possível **criar** um saque técnico de R$ 10 (status `pendente`) sem usar o saldo Asaas — usa wallet interna.

### B) Concluir homologação PIX OUT + auth webhook (uso do saldo Asaas)

| Critério | Avaliação |
|----------|-----------|
| `net_amount` mínimo (regras atuais) | R$ 8,00 |
| Saldo Asaas disponível | ≈ R$ 5,19 |
| Gap | **R$ 2,81** (sem margem para `transferFee` Asaas) |

## 3.2 Resposta final Fase 3

**É possível criar um saque técnico válido utilizando esse saldo (R$ 5,19 Asaas) para concluir a homologação?**

# **Não**

**Regra que impede:** sob a configuração vigente, o menor PIX OUT real é **R$ 8,00 líquido** (saque R$ 10 − taxa R$ 2). O saldo Asaas ≈ R$ 5,19 é **inferior** ao `net_amount` mínimo operacional. A execução falharia na API Asaas (saldo insuficiente / `invalid_action`), e o webhook de autorização não completaria o ciclo de homologação pretendido.

**Regras envolvidas:**

1. `minWithdrawAmount = 10.00` → `net_amount = 8.00` (`server-fly.js:1667–1797`)
2. `createPixTransfer` envia `value = net_amount` (`asaas-http-client.js:593–631`)
3. Saldo Asaas < R$ 8,00 → transferência não executável

# FASE 4 — Configuração existente para homologação < R$ 10

Auditoria de bypasses, flags e modos que permitiriam homologação com valor inferior a R$ 10 **sem alterar código**:

| Mecanismo | Permite < R$ 10? | Evidência |
|-----------|------------------|-----------|
| Variável ENV mínimo saque | **Não existe** | Grep em codebase — sem `MIN_WITHDRAW*` |
| `PAGAMENTO_TAXA_SAQUE=0` no Fly | Reduz líquido para R$ 10, **não** abaixo do mínimo bruto | Ainda exige R$ 10 Asaas |
| `P16B_MIN_AMOUNT` / `P15Y_MIN_AMOUNT` | Só filtra scripts | Não altera `POST /api/withdraw/request` |
| Modo desenvolvimento / sandbox | Sandbox min Asaas = R$ 5, mas homologação real exige produção | `SANDBOX_MIN_PIX_VALUE = 5` — gates prod OFF |
| Conta técnica | Classificação read-only | `technicalE2EAccount.js` — não altera limites |
| Bypass homologação auth | **Não encontrado** | Auth webhook não tem flag de valor |
| Config Asaas produção | Min API = **R$ 1** | `PRODUCTION_MIN_PIX_VALUE = 1` — **não alcançável** pelo fluxo Gol de Ouro |
| Admin `approve-and-send` | Usa `net_amount` do saque existente | Não contorna mínimo na criação |
| Inserção manual DB | Possível tecnicamente | **Fora de escopo** — proibido nesta auditoria |

### Conclusão Fase 4

**Não existe configuração deployada hoje que permita homologação PIX OUT real com valor inferior a R$ 10 bruto (R$ 8 líquido)** sem alterar código ou regras de negócio.

A única flexibilidade existente é no **provider Asaas** (mín. R$ 1), mas o **backend Gol de Ouro** impõe R$ 10 antes do provider ser chamado.

## Estratégia recomendada — Opção B (com preparação P1.6B.R3)

### Passo 1 — Funding Asaas (read-only até aqui; ação operacional futura)

Depositar na conta Asaas produção valor suficiente para cobrir:

- `net_amount` mínimo: **R$ 8,00**
- Margem operacional: **+R$ 1–2** (buffer `transferFee` / arredondamentos)
- **Total sugerido:** depositar **≥ R$ 3,00** (atinge ~R$ 8,19+) ou **≥ R$ 5,00** (margem confortável → ~R$ 10,19)

### Passo 2 — Criar saque técnico (fora desta auditoria)

1. Autenticar conta técnica `85872488-9e4c-42df-8978-7f9ef9f5cb00`
2. `POST /api/withdraw/request` — **R$ 10,00**, chave PIX válida
3. Confirmar status `pendente` + `net_amount = 8.00`
4. `node scripts/p16b-r3-audit-saques.mjs` → esperar `READY`

### Passo 3 — Homologação P1.6C (janela única)

1. Confirmar painel Asaas: URL + token auth webhook (P1.6B.R2)
2. Abrir gates PIX OUT (janela controlada)
3. Uma execução `approve-and-send` / `P15Y_PHASE=execute`
4. Capturar logs `[ASAAS][TRANSFER_AUTH]` → `APPROVED`
5. Fechar gates imediatamente

### Por que não reduzir o mínimo para R$ 3 (líquido R$ 1)?

Teoricamente R$ 5,19 Asaas cobriria PIX de R$ 1, mas exigiria **alteração de código** em `server-fly.js:1667`, `paymentController.js:297` e `Withdraw.jsx:27` — viola política de zero regressão e é desnecessário frente a um depósito de ~R$ 3.

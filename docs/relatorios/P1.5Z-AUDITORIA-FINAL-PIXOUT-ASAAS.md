## Veredito Final

# **PASS COM RESSALVAS**

Não foram encontradas evidências de **falha estrutural** na Indesconectável Payment Engine™ que impeça o PIX OUT. A requisição percorreu corretamente toda a cadeia interna e foi **rejeitada pelo Asaas** com HTTP **400** e `errorCode: **invalid_action**`. O bloqueio é atribuível, com **alta confiança**, a **limitação operacional da conta Asaas** (autorização crítica / transferências API não liberadas para automação), com **saldo insuficiente** como fator secundário possível.

**Ressalvas técnicas objetivas (não bloqueiam veredito PASS):**

| Ressalva | Natureza |
|----------|----------|
| Logs não persistem `description` completa do erro Asaas | Observabilidade |
| Rollback da taxa (R$ 2) incompleto no saque R2 | Ledger interno |
| `asaas_transfer_status=DONE` com `transfer_id` null | Higiene de persistência + replay de teste pós-R2 |
| Saldo/KYC Asaas não confirmados via API read-only | Lacuna de evidência externa |

## Resumo Executivo

Durante P1.5Y.EXEC.R2, o fluxo completo Admin → Compat → Factory → Asaas Provider → HTTP Client → `POST https://api.asaas.com/v3/transfers` foi executado em **produção** com gates abertos. O Asaas respondeu em ~432 ms com **HTTP 400** e **`invalid_action`**. Nenhum `transfer_id` foi criado. A engine tratou a falha, registrou `falha_payout`, executou rollback parcial e fechou os gates.

**Mudança arquitetural confirmada:** o erro **não ocorre** dentro da Payment Engine — ocorre na **resposta do provedor**.

| Critério de aceitação | Resposta |
|----------------------|----------|
| 1. Engine construiu corretamente a requisição? | **SIM** |
| 2. Autenticação funcionando? | **SIM** (não houve 401/403) |
| 3. Endpoint correto? | **SIM** |
| 4. Payload compatível com doc Asaas? | **SIM** |
| 5. Evidência de falha estrutural na engine? | **NÃO** |
| 6. `invalid_action` atribuível a limitação conta/provedor? | **SIM** (alta confiança) |

**Nível de confiança da conclusão:** **85%** — limitado pela ausência do body completo do erro em logs e por não ter sido executado `GET /myAccount` / `GET /finance/balance` em produção nesta auditoria.

## Conclusão Técnica

A Indesconectável Payment Engine™ **cumpriu sua função** na homologação P1.5Y.EXEC.R2:

1. Selecionou o provider Asaas corretamente.
2. Construiu payload conforme contrato OpenAPI.
3. Autenticou e enviou `POST /v3/transfers` em produção.
4. Recebeu rejeição do **provedor** (`invalid_action`).
5. Propagou erro, registrou ledger de falha e executou rollback sem criar transferência.

O erro **migrou da camada interna para a camada do provedor** — exatamente como previsto quando a integração técnica está madura mas a conta não está habilitada para automação.

**Não há evidência** de que corrigir código da engine resolveria o `invalid_action` sem ação no lado Asaas (suporte, auth crítica, saldo, habilitação).

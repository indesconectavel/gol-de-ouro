# P1.5ZZ — Encerramento Técnico da Validação da Indesconectável Payment Engine™

**Projeto:** Gol de Ouro™ V1  
**Arquitetura:** Indesconectável Payment Engine™  
**Classificação:** Documento técnico institucional de encerramento  
**Data:** 2026-06-29  
**Modo:** Documental — sem alteração de código, banco, gates ou runtime

---

## 1. Resumo Executivo

A **Indesconectável Payment Engine™** atingiu o ponto de **validação técnica esperado para a trilha P1.5** do Gol de Ouro™ V1.

Entre as etapas P1.5A e P1.5I, a arquitetura financeira foi evoluída de um estado em que o PIX OUT Asaas em produção **não existia** (P1.5A — FAIL) para uma engine **provider-aware**, com gates, HTTP client, schema, webhooks, reconciliação ledger/wallet e homologação operacional controlada em ambiente mock.

A janela operacional **P1.5Y.EXEC.R2** (2026-06-29) comprovou que o fluxo completo percorre a cadeia interna até o provedor Asaas em produção. A API respondeu **HTTP 400** com `errorCode: invalid_action`. **Nenhuma transferência foi criada.** A engine tratou a falha com rollback e fechamento de gates.

As auditorias **P1.5Z** (PASS COM RESSALVAS) e **P1.5FIN** (FIN-PASS-COM-RESSALVAS) consolidaram que:

- **não há evidência objetiva de falha estrutural** na Payment Engine;
- o bloqueio remanescente do PIX OUT Asaas concentra-se na **camada operacional/regulatória do provedor ou da conta**, não na construção interna da requisição.

Este documento **formaliza o encerramento técnico** da trilha P1.5. Nova tentativa operacional de PIX OUT real depende de **confirmação oficial do Asaas** e **autorização explícita** do operador.

## 5. Evidência Central

### Fluxo comprovado em produção (P1.5Y.EXEC.R2)

```text
Admin (approveAndSendWithdrawAdmin)
  ↓
Compat Layer (createPixWithdrawCompat)
  ↓
FinanceProviderFactory (PAYOUT_PROVIDER=asaas)
  ↓
Asaas Provider (createPixWithdraw)
  ↓
HTTP Client (createPixTransfer, httpGate: pixOutProduction)
  ↓
POST https://api.asaas.com/v3/transfers
  ↓
Asaas Produção
  ↓
HTTP 400 — errorCode: invalid_action
```

### Interpretação

| Fato | Implicação |
|------|------------|
| Logs `asaas_api_request_start` → `asaas_api_request_failed` | Requisição **saiu da aplicação** e **chegou ao Asaas** |
| HTTP 400 (não 401/403/404/5xx) | Autenticação e roteamento **aceitos**; rejeição de **negócio/conta** |
| `transfer_id` null | Nenhum PIX OUT realizado |
| Rollback + gates fechados | Patrimônio interno **protegido** |

| Hipótese | Status | Justificativa |

| Falha na Compat Layer | **Refutada** | Fluxo chegou ao provider; request HTTP executado |
| Falha no Asaas Provider | **Refutada** | Endpoint atingido; body conforme OpenAPI (P1.5Z) |
| Falha no HTTP Client | **Refutada** | POST executado; resposta 400 parseada em ~432 ms |
| Payload incompatível | **Refutada** | P1.5Z: compatível; F4.2G sandbox mesmo shape funcionou |
| Autenticação inválida | **Refutada** | Erro não foi 401/403; API Key produção processada |
| Endpoint incorreto | **Refutada** | `POST /v3/transfers` oficial documentado Asaas |
| Restrição operacional Asaas | **Hipótese predominante** | `invalid_action` retornado pelo provedor; doc Asaas associa a autorização crítica |
| Restrição regulatória/KYC | **Possível (inconclusiva)** | Depende de confirmação oficial Asaas — não comprovada nesta trilha |
| Saldo insuficiente | **Possível fator secundário** | Print operador ~R$ 3,02 vs R$ 8,00 tentativa; não altera conclusão principal (P1.5Z) |

## 7. Conclusão Técnica

Com base nas evidências consolidadas das auditorias **P1.5Z** e **P1.5FIN**:

> **Não há evidência objetiva de falha estrutural na Indesconectável Payment Engine™** que impeça o funcionamento correto do PIX OUT do ponto de vista de arquitetura, roteamento, autenticação, endpoint ou construção de payload.

O bloqueio remanescente do PIX OUT Asaas em produção deve ser tratado como **dependência externa de provedor/conta**, exigindo:

1. Confirmação oficial do Asaas sobre o significado de `invalid_action` na conta utilizada;
2. Habilitação operacional para transferências PIX via API (autorização crítica, IP whitelist, webhook de autenticação ou equivalente);
3. Saldo e limites adequados para nova homologação controlada.

A engine cumpriu o papel de **isolar, proteger e rastrear** a tentativa — incluindo rollback e fechamento de gates.

## 10. Próximas Ações Recomendadas

1. **Abrir chamado técnico no Asaas** solicitando explicação objetiva para `invalid_action` em `POST /v3/transfers`.
2. **Enviar evidências:** timestamp UTC `2026-06-29T05:28:26Z`, endpoint, HTTP 400, `externalReference` `wd_67786aba15ef420083118834` (sem API Key, CPF ou chave PIX completos).
3. **Perguntar** se a conta possui Transfer API **efetivamente habilitada** para automação sem confirmação manual.
4. **Confirmar** exigências regulatórias/KYC para PIX OUT via API.
5. **Confirmar** exigência de saldo mínimo (tentativa R2: R$ 8,00 líquido).
6. **Não alterar a engine** até retorno oficial do provedor.
7. **Após retorno do Asaas**, decidir entre:
   - nova homologação controlada (P1.5Y.EXEC.R3);
   - ajuste técnico pontual (se suporte indicar divergência de contrato);
   - troca ou adição de PSP para PIX OUT;
   - congelamento da integração Asaas OUT até habilitação formal.
